## Architecture Limitations

### 1. Client-Side Price Fetching

- **Issue**: Gold prices and FX rates are fetched directly from the browser. Each visitor makes
  their own API calls to goldpricez.com and exchangerate-api.com.
- **Impact**: API rate limits may be hit with high traffic. Each visitor experiences their own
  latency for price loading.
- **Mitigation**: Dual-layer localStorage cache reduces redundant API calls. Service worker caches
  responses network-first.
- **Ideal fix**: Server-side price caching proxy that refreshes prices centrally and serves them to
  all visitors.

### 2. Static Page Generation

- **Issue**: Country/city/karat leaf pages (~400 files) are pre-generated by
  `build/generatePages.js` and committed to the repo. Adding a new country or city requires
  regenerating all affected pages.
- **Impact**: Repository size grows with each country. Page content is identical except for metadata
  — actual prices are fetched live.
- **Mitigation**: The `page-hydrator.js` module handles live data injection at runtime.
- **Ideal fix**: Server-side rendering or a single template page with client-side routing.

### 3. No Server-Side Rendering

- **Issue**: All pages are static HTML. Prices shown to search engine crawlers are placeholder
  values or empty, not live data.
- **Impact**: Search engines don't index actual gold prices. Rich snippets with prices won't show in
  search results.
- **Mitigation**: Structured data (JSON-LD) is present but with static/generic values.
- **Ideal fix**: SSR or pre-rendering with actual price data at build time.

### 4. Dual URL Structure (Legacy + Hierarchical)

- **Issue**: Country pages exist at both `/countries/uae/` (summary) and `/uae/gold-price/`
  (hierarchical). Both serve similar content.
- **Impact**: Potential duplicate content, split link equity, confusing for developers.
- **Mitigation**: Canonical tags and .htaccess redirects cover some paths.
- **Ideal fix**: Full migration to hierarchical structure with 301 redirects from legacy URLs.

### 5. GitHub Pages Cannot Set HTTP Headers

- **Issue**: GitHub Pages does not support custom HTTP headers (CSP, HSTS, etc.).
- **Impact**: Security headers are limited to `<meta>` equivalents, which have reduced effectiveness
  compared to HTTP headers.
- **Mitigation**: Meta-equivalent headers for `X-Content-Type-Options`, `X-Frame-Options`,
  `Referrer-Policy`. Express server adds proper headers when used.
- **Ideal fix**: Deploy to a platform that supports custom headers (Vercel, Netlify, Cloudflare
  Pages) or use the Express server.

## Performance Bottlenecks

### 1. Large CSS Bundle

- **Issue**: `styles/global.css` is 4,548 lines — a single monolithic file containing global styles,
  component styles, and responsive breakpoints.
- **Impact**: Every page loads the entire CSS file even if it only uses a fraction.
- **Mitigation**: Vite minifies CSS in production. Page-specific CSS files exist for additional
  styles.
- **Ideal fix**: Split `styles/global.css` into smaller modules; use CSS containment or
  code-splitting.

### 2. No Image Optimization Pipeline

- **Issue**: No automated image optimization, no WebP/AVIF conversion, no responsive `srcset`.
- **Impact**: Images (though few) are served at original size.
- **Mitigation**: The site is mostly text-based with minimal images.
- **Ideal fix**: Add image optimization to build pipeline if more images are added.

### 3. No Lazy Loading

- **Issue**: No `loading="lazy"` attributes on images or iframes.
- **Impact**: Negligible currently (few images), but would matter if more visual content is added.
- **Ideal fix**: Add `loading="lazy"` to below-the-fold images.

### 4. External Script Dependencies

- **Issue**: Admin page loads Leaflet.js and Chart.js from CDNs (unpkg, jsdelivr).
- **Impact**: Additional HTTP requests and potential CDN availability issues.
- **Mitigation**: Admin pages are not public-facing and are not performance-critical.
- **Ideal fix**: Bundle or self-host critical dependencies.

### 5. API Waterfall on Page Load

- **Issue**: Price fetching happens sequentially — gold price first, then FX rates, then
  calculations, then rendering.
- **Impact**: Visible loading delay before prices appear.
- **Mitigation**: Skeleton loading states provide visual feedback. Cached values show immediately.
- **Ideal fix**: Parallel API fetching and progressive rendering.

## UX Issues

### 1. No Error Recovery UI

- **Issue**: When API calls fail and there's no cached data, users see empty/broken state.
- **Impact**: First-time visitors without cache have no fallback experience.
- **Mitigation**: Stale data badges are shown when cached data is used.
- **Ideal fix**: Better empty states with retry buttons and informational messaging.

### 2. Limited Accessibility Testing

- **Issue**: While semantic HTML and ARIA labels are used, there's no automated accessibility
  testing in CI.
- **Impact**: Accessibility regressions may not be caught.
- **Mitigation**: Manual attention to contrast, focus states, and keyboard navigation.
- **Ideal fix**: Add axe-core or pa11y to CI pipeline.

### 3. Chart Interaction on Mobile

- **Issue**: The price chart uses pointer events that may not translate well to touch on all mobile
  devices.
- **Impact**: Some touch interactions may be imprecise.
- **Mitigation**: Basic touch support is present.
- **Ideal fix**: Test and optimize chart touch interactions on various devices.

### 4. Navigation Depth on Mobile

- **Issue**: The navigation has 6 dropdown groups with many items. On mobile, this creates a very
  long scrolling menu.
- **Impact**: Users may have difficulty finding specific items in the mobile menu.
- **Ideal fix**: Consider collapsible accordion groups or a search-within-menu feature.

## SEO Weaknesses

### 1. Missing Meta Descriptions

- **Pages missing descriptions**: `terms.html`, `privacy.html`
- **Impact**: Search engines may generate their own snippets, which may not be optimal.
- **Fix effort**: Low — add `<meta name="description">` tags.

### 2. Static Price Data in Structured Data

- **Issue**: JSON-LD structured data on leaf pages may contain static/placeholder prices rather than
  live values.
- **Impact**: Rich snippets in search results may show stale prices.
- **Ideal fix**: Server-side rendering to inject live prices into structured data at request time.

### 3. Duplicate Content Risk

- **Issue**: Legacy country pages and hierarchical pages serve similar content.
- **Impact**: Search engines may index both, splitting page authority.
- **Mitigation**: Canonical tags are present on most pages.
- **Ideal fix**: 301 redirects from legacy to hierarchical URLs.

### 4. No XML Sitemap Auto-Generation in CI

- **Issue**: Sitemap is generated manually or semi-manually. It could drift from actual pages.
- **Impact**: Search engines might not discover new pages quickly.
- **Ideal fix**: Add sitemap generation to the deploy workflow.

### 5. No hreflang on All Pages

- **Issue**: Not all pages have `hreflang` tags for English/Arabic language alternates.
- **Impact**: Search engines may not properly associate language variants.
- **Ideal fix**: Add `hreflang` tags to all bilingual pages.

## Security Considerations

### 1. Supabase Anon Key in Client Code

- **Issue**: `src/config/supabase.js` contains the Supabase URL and anon key in committed source.
- **Impact**: Low — anon keys are designed for client-side use with RLS.
- **Mitigation**: Row Level Security ensures anon users can only read verified shops and settings.

### 2. No Content Security Policy via HTTP Headers

- **Issue**: CSP is only set via `<meta>` tags on GitHub Pages, which has limitations.
- **Impact**: Inline script injection protection is weaker than HTTP header CSP.
- **Mitigation**: The Express server has full Helmet CSP configuration.

### 3. AdSense Placeholder

- **Issue**: Placeholder publisher ID is in code. When replaced with real ID, ad scripts will
  execute.
- **Impact**: None currently. Ensure CSP is updated when ads are activated.

## Scalability Constraints

### 1. File-Based Repository Growth

- **Issue**: Each new country adds ~30 generated HTML files to the repo. With 15 countries, there
  are already 400+ HTML files.
- **Impact**: Repository clone time and disk usage increase linearly.
- **Ideal fix**: Template-based rendering instead of pre-generated files.

### 2. Client-Side API Rate Limits

- **Issue**: Free-tier APIs (goldpricez.com, exchangerate-api.com) have rate limits.
- **Impact**: High traffic could exhaust API quotas, causing price display failures.
- **Ideal fix**: Server-side caching proxy with a shared cache for all visitors.

### 3. No Database for Public Data

- **Issue**: Public shop data is hardcoded in `data/shops.js`. Adding/editing shops requires code
  changes.
- **Impact**: Non-technical users cannot manage shop data without developer help.
- **Mitigation**: Admin panel with Supabase integration exists for verified shops.

## Developer Experience

### 1. Vite/Node.js Version Compatibility

- **Issue**: Vite 8.x may not work with the latest Node.js versions (v24+).
- **Impact**: Developers on bleeding-edge Node.js cannot run `npm run build`.
- **Mitigation**: CI uses pinned Node.js version.
- **Ideal fix**: Document required Node.js version; add `.nvmrc` file.

### 2. No .env.example File

- **Issue**: No template `.env` file showing required environment variables.
- **Impact**: New developers must read docs to know which vars to set.
- **Mitigation**: `docs/environment-variables.md` documents all variables.
- **Ideal fix**: Add `.env.example` to repo root.

### 3. Large Style.css File

- **Issue**: 4,548-line CSS file is difficult to navigate and maintain.
- **Impact**: Finding specific styles requires extensive searching.
- **Ideal fix**: Split into modular CSS files by component/section.

### 4. No Type Checking

- **Issue**: Pure JavaScript without TypeScript or JSDoc type annotations in most files.
- **Impact**: IDE support is limited; type errors are caught at runtime only.
- **Mitigation**: ESLint catches many common errors.
- **Ideal fix**: Add JSDoc annotations or migrate to TypeScript incrementally.
