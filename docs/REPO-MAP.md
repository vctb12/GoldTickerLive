## 1. Page generation & routing

- **Vite multi-page input → two tiers**: `vite.config.js:26–55` uses `discoverHtmlEntries()` to
  recursively scan the root for `.html` files, excluding `countries/`, `admin/`, and `embed/`
  directories. Each file becomes an entry point with a dash-joined key (e.g., `index-html` for
  `/index.html`, `scripts-node-index-html` for `/scripts/node/index.html`). Excluded directories are
  served verbatim (copied by deploy workflow) and never processed by Vite.

- **Generated country/city/karat pages**: `build/generatePages.js` is the primary page generator,
  run as a build step (not invoked by Vite). It parses `src/config/countries.js` and
  `src/config/karats.js` using safe VM evaluation (`evalConfigArray`), then generates three types of
  static HTML per country-city combination: (1) `{country}/gold-price/index.html` (country hub—all
  karats in one page), (2) `{country}/{city}/gold-rate/index.html` (city hub—all 4 karats: 24K, 22K,
  21K, 18K), (3) `{country}/{city}/gold-shops/index.html` (directory listing). For cities,
  karat-specific pages are also generated at `{country}/{city}/gold-rate/{karat-slug}/index.html`
  (lines 263–288).

- **Page hydration model**: All generated country/city/karat pages include a skeleton HTML structure
  (lines 62–118 in `generatePages.js`) with placeholders for nav/footer/breadcrumbs and a
  `<div id="price-display">` that is populated by client-side code. The hydrator is loaded via
  `<script type="module" src="../../src/lib/page-hydrator.js">` (relative depth-based path).
  `src/lib/page-hydrator.js` fetches live prices from the API, renders karat cards, breadcrumbs, and
  related links dynamically—this is why all generated pages share the same minimal HTML skeleton but
  display different content per route.

- **Hand-authored vs. generated pages (5 root HTML files + 3 generated types +
  content/admin/redesign trees)**:
  - **Root `.html` files** (served at top-level, bundled by Vite): `index.html`, `tracker.html`,
    `calculator.html`, `shops.html`, `learn.html`, `insights.html`, `invest.html`,
    `methodology.html`, `pricing.html`, `account.html`, `compare.html`, `dashboard.html`,
    `designer-lab.html`, `developer.html`, `offline.html`, `privacy.html`, `terms.html`, `404.html`.
    These are hand-authored entry points.
  - **Generated tree** (`countries/`): 26 countries × (1 country hub + N cities × 3 pages per city
    [gold-rate + gold-shops + karat-specifics]) = ~390 pages total. Example: UAE has 7 cities, each
    with gold-rate, gold-shops, and 4 karat pages = 35 pages just for UAE.
  - **Content pages** (`content/guides/`, `content/gold-price-history/`, `content/faq/`,
    `content/submit-shop/`, etc.): Hand-authored HTML files, processed by Vite as regular entries.
  - **Admin** (`admin/`): Separate SPA with top-level `admin/index.html` plus sub-modules
    (analytics, content, leads, etc.). Uses Supabase auth and top-level await; excluded from Vite
    bundling (line 17 in vite.config.js).
  - **Redesign** (`redesign/`): Documentation/planning files (AUDIT.md, DESIGN-SYSTEM.md,
    PHASE2-PLAN.md); not part of the build.

- **City hub pages (noindex navigation stubs)**: Each city at `{country}/{city}/index.html` is a
  lightweight stub page (e.g., `/countries/uae/dubai/index.html`, 5 KB), not generated by
  `generatePages.js`. It uses `stub-city.css` (`countries/stub-city.css`) for minimal styling and
  boots via `src/lib/content-page-boot.js`. The stub has
  `<meta name="robots" content="noindex,follow">` (visible in `/countries/uae/dubai/index.html:25`)
  to prevent indexing the navigation page itself. Its purpose is to guide users to either
  `gold-rate/` (live prices) or `gold-shops/` (directory).

- **Chart & gold-price & methodology directories**: Empty; they serve as URL prefixes in the routing
  layer only (not HTML entry points). Examples: `/chart/`, `/gold-price/24k/` are likely redirected
  or aliased via `_redirects`.

- **URL routing via \_redirects (Netlify/Cloudflare)**: `_redirects` file (148 lines) maps clean
  URLs to `.html` files and manages legacy aliases:
  - Clean URLs (301 permanent): `/tracker` → `/tracker.html`, `/calculator` → `/calculator.html`,
    etc. (lines 19–34)
  - Country/city shorthand aliases: `/uae/` → `/countries/uae/`, `/dubai` → `/countries/uae/` (lines
    38–64)
  - Consolidation redirects: `/content/guides/` → `/learn.html`, `/content/tools/` →
    `/calculator.html`, etc. (lines 85–110)
  - City URL consolidation: `/countries/:country/:city/gold-prices/` →
    `/countries/:country/:city/gold-rate/` (lines 113–116)
  - Country gold-price deduplication: `/countries/:country/gold-price/` → `/countries/:country/`
    (lines 119–120)
  - Catch-all 404 (line 148): `/*` → `/404.html` with 404 status (NOT a 200 soft-redirect).

- **404.html (hard 404 fallback)**: `/404.html` is a static page (14 KB) that renders a search box,
  primary links (tracker, calculator, countries, search), secondary chip navigation, and a live
  reference price pill fetched from cache. It loads `/src/pages/not-found.js` to hydrate the
  freshness indicator. The `_redirects` file does NOT use a catch-all soft-redirect (which would
  hide true 404s from crawlers); instead, it allows unknown URLs to fall through to the 404.html
  handler with a legitimate 404 status code (line 148).

- **\_headers (static caching & security)**: Sets cache headers: immutable long-term (31536000s) for
  fingerprinted assets (`/assets/*`, `/styles/*`, `/src/*`), short-term with must-revalidate (300s)
  for HTML pages, and CSP/security headers (X-Content-Type-Options, X-Frame-Options,
  Strict-Transport-Security, etc.) applied globally. Admin panel marked as `noindex, no-store`.

- **CNAME file**: Single line `goldtickerlive.com` (custom domain for GitHub Pages / Netlify /
  Cloudflare Pages deployment).

- **Sitemap generation**: `build/generateSitemap.js` (auto-run during `npm run build`) generates
  `sitemap.xml` by: (1) including all static pages with priorities (home=1.0, tracker=0.95, etc.),
  (2) iterating all COUNTRIES and KARATS, (3) emitting URLs for country hubs (0.85 priority), city
  hubs (0.80 priority), city gold-rate hubs (0.80 priority), and individual karat pages (0.70
  priority). Each entry includes hreflang alternates for `x-default`, `en`, and `ar?lang=ar` to
  signal language variants to crawlers. URLs flagged with `noindex` meta tag (e.g., city nav stubs)
  are excluded from sitemap (lines 16–23).

- **Build pipeline**: `npm run build` orchestrates: `extract-baseline.js` → `normalize-shops.js` →
  `render-learn-static-fallback.mjs` → `inject-theme-preinit.js` → `inject-schema.js` →
  `generateSitemap.js` → `vite build`. This sequence ensures generated pages exist in the source
  tree before Vite sees them, allows pre-processing scripts to enrich HTML, and produces a dist/
  folder ready to deploy to a static host.

- **Breadcrumb & navigation injection**: Generated pages contain empty root divs (`id="nav-root"`,
  `id="breadcrumb-root"`, `id="footer-root"`, line 93–114 in `generatePages.js`) that are populated
  client-side by `page-hydrator.js`, which imports `injectNav()`, `injectBreadcrumbs()`, and
  `injectFooter()` from `src/components/nav.js`, `src/components/breadcrumbs.js`, and
  `src/components/footer.js`. This decouples static HTML structure from dynamic chrome.

- **Language alternation**: All generated pages include hreflang tags (e.g.,
  `<link rel="alternate" hreflang="ar" href="{canonical}/?lang=ar">` in line 80 of
  `generatePages.js`). The `page-hydrator.js` honors `?lang=ar` query parameter and reads user lang
  preference from localStorage (`CONSTANTS.CACHE_KEYS.userPrefs`) to determine initial language,
  then persists the choice (lines 72–91 in `page-hydrator.js`).

- **Page depth calculation**: Relative CSS/JS paths in generated pages depend on directory depth.
  `relPrefix(depth)` utility (line 48 in `generatePages.js`) outputs `../` repeated depth times.
  Country hubs are at depth 2 (`{country}/gold-price/`), city hubs at depth 3
  (`{country}/{city}/gold-rate/`), karat-specific at depth 4
  (`{country}/{city}/gold-rate/{karat}/`), ensuring all relative imports resolve correctly.

- **Related links on generated pages**: Generated pages include `relatedLinks` HTML (line 111 in
  `generatePages.js`). For country hubs, this lists all cities as chips. For city hubs, this
  includes links to: the country overview, gold shops, and sibling cities (max 4, line 212). For
  karat pages, this links back to the city's all-karats hub. This creates a navigation mesh within
  the generated tree.

- **JSON-LD schema**: All generated pages include structured data. `buildPriceLd()` (lines 124–147)
  creates a BreadcrumbList schema with position-numbered items mapping labels to canonical URLs.
  Karat pages include additional Dataset schema with `dateModified` and `distribution` metadata for
  pricing data (lines 240–260 in `generatePages.js`). This signals to search engines that the page
  contains time-series pricing information.

**Key files:** `vite.config.js`, `build/generatePages.js`, `build/generateSitemap.js`, `_redirects`,
`_headers`, `404.html`, `src/lib/page-hydrator.js`, `src/config/countries.js`,
`countries/stub-city.css`, `CNAME`

## 2. Shared Shell: Nav / Header / Footer / Ticker Architecture

**Orchestration & Core Injection**

- All shell surfaces (nav, footer, ticker, spotBar) orchestrated through a single command:
  `mountSharedShell()` → `src/components/site-shell.js:10-29`. Takes `{lang, depth, withSpotBar}`
  options, calls individual injectors in sequence, returns shell controller with `updateLang()`
  support. **Why it matters**: single point of control eliminates initialization race conditions and
  ensures all surfaces share lang/depth state.
- Individual injectors (`injectNav`, `injectFooter`, `injectTicker`, `injectSpotBar`) guard against
  double-injection by checking for existing DOM elements (.site-nav, #gold-ticker, etc.). Prevents
  accidental duplication on long-lived pages or test reruns.

**Navigation & Header**

- **Core markup generation**: `src/components/nav.js:329-499` builds 2-tier nav (desktop mega-menu
  dropdowns + mobile off-canvas drawer + search overlay + theme/lang toggles + CTA). All content
  table-driven from `NAV_DATA[lang]` in `nav-data.js:1-100+`.
- **Page shell styling bridge**: `applyPageShell()` (nav.js:131-145) sets `data-page`,
  `data-section`, `data-shell`, `data-accent` attrs on `<body>` and nav element. CSS uses these to
  vary header accent color per-section (Prices→amber, Learn→indigo, etc.). Computed from current URL
  via `getPageShell()` which pattern-matches `PAGE_SHELLS` array.
- **Active state matching**: `isPageMatch()` (nav.js:87-112) uses hardened path logic: strips
  relative `../` prefixes, resolves to absolute paths, matches against `location.pathname`. Root `/`
  only matches homepage exactly. No false positives across depth levels.
- **Mobile bottom nav**: 5-button quickbar injected after header
  (Tracker/Calc/Countries/Shops/Menu). Wired to drawer menu toggle via data attributes. Language
  labels live-updated.
- **Scroll hide/reveal behavior**: `_initNavScrollBehavior()` (nav.js:909-965) tracks scroll delta,
  hides nav when scrolling down >6px, reveals when scrolling up or near top (< 64px). Respects
  prefers-reduced-motion. Prevents nav-hidden state while drawer/dropdown open.

**Theme Preinit (Dark/Light Flash Prevention)**

- **Render-blocking script injection**: `scripts/node/inject-theme-preinit.js` idempotently injects
  a 50-byte inline `<script>` into every HTML `<head>` (right after `<meta charset>`) to set
  `data-theme` + `data-theme-mode` BEFORE first paint.
- **Mirrors nav.js logic exactly**: reads `localStorage.getItem('user_prefs').theme` (values:
  'auto'|'light'|'dark'), resolves 'auto' via `prefers-color-scheme: dark` media query, sets
  `<html>` attrs. Nav.js later applies identical attrs with no repaint.
- **Idempotent by marker**: skips pages already containing `gtl-theme-preinit` marker comment.
  Excludes `/admin` dir (separate internal shell), `/content/embed/` fragments, and
  non-full-document partials.
- **CI validation**: `--check` flag fails the build if any public page is missing the preinit script
  (enforced in CI).

**Footer**

- **5-column layout**: `src/components/footer.js:20-193` renders brand column (tagline + badges),
  then 4 link columns (each from a `NAV_DATA.groups` section, max 3 links per section to keep mobile
  readable).
- **Newsletter subscription**: Built-in form with dual endpoints (internal
  `/api/v1/newsletter/subscribe` or legacy Formspree). Full state machine: loading →
  success|duplicate|invalid|error. Email validation + honeypot field. Tracks via analytics.
- **Data attribution + freshness**: Displays price source URL (gold-api.com), FX source
  (open.er-api.com), gold data freshness timestamp (read from `localStorage.goldprices_gold_ts`),
  data sync timestamp (from `localStorage.goldUpdatedAt`).
- **Fixed 3.6725 AED/USD peg link** + Troy oz definition (31.1035g) + disclaimer (spot-linked
  reference only, not financial advice).
- **Admin access Easter eggs** (3 methods): triple-click ⚡ footer symbol, Ctrl+Shift+A keyboard
  shortcut, or Konami code (↑↑↓↓←→←→BA). Shows a styled popup with admin link + 8-second
  auto-dismiss.

**Live Price Ticker (Bottom)**

- **Component**: `src/components/ticker.js:120-217` creates a fixed-bottom smooth-scrolling marquee
  showing XAU/USD + 4 UAE rates (24K/22K/21K/18K in AED/gram).
- **Dual-copy animation**: Two identical ticker-copy divs for seamless infinite scroll; CSS
  animation loops them horizontally.
- **Freshness pill**: Status indicator (● dot + label) shows freshness key
  ('live'|'cached'|'stale'|'unavailable') with bilingual labels. Tooltip includes timestamp +
  relative age (e.g., "2 min ago").
- **Honors AGENTS.md rule 2** (non-negotiable: freshness must be visible): `updateTicker(data)`
  recomputes freshness from `updatedAt` timestamp via `getLiveFreshness()`. Never silently regresses
  to 'live' without a fresh timestamp.
- **UX polish**: Pause animation on hover (desktop only), respect prefers-reduced-motion (disables
  scroll, makes overflow auto), dismiss button persists state to sessionStorage, links point to
  tracker.html.

**Spot Bar (Top, Optional)**

- **Component**: `src/components/spotBar.js:37-108` injects sticky bar above nav (unless hidden via
  `withSpotBar: false`) showing XAU/USD + 24K AED/gram with freshness timestamp.
- **Freshness badge**: Same system as ticker. Displays clock time + relative age (e.g., "12:34 · 1
  min ago"). Tooltip = freshness key label.
- **Dynamic height tracking**: ResizeObserver (or resize event fallback) measures bar height, sets
  `--spot-bar-height` CSS var so page content doesn't jitter when bar appears/disappears.
- **Price change animation**: Tracks previous XAU/USD value, applies CSS classes
  (spot-change-up/down/neutral) for 200ms to flash on delta.
- **Positioned relative to nav**: `insertBefore(nav)` if nav exists, else
  `insertBefore(body.firstChild)`.

**Search Integration**

- **Lazy-loaded engine**: `src/search/searchEngine.js` imports `SEARCH_INDEX` from `searchIndex.js`.
  Scoring algorithm: exact > starts-with > contains > keyword > fuzzy (Levenshtein with 1-char
  tolerance for 3+ char queries).
- **Nav search UI**: `initNavSearch()` (nav.js:1308-1561) wires global keyboard shortcuts (`/` and
  Ctrl/Cmd+K) to open search overlay. Debounced input (200ms) triggers lazy-load of search module,
  renders grouped results by type (Countries/Cities/Pages/Karats/Shops).
- **Recent searches**: Persisted in `localStorage.nav.search.recent` (cap 8 items). Shown when input
  empty. Cleared by ESC key.
- **Accessible dropdowns**: Arrow up/down for keyboard nav through results, Enter to navigate, Tab
  to close overlay.

**Language Switching**

- **Live update without re-inject**: `updateNavLang()` (nav.js:1089-1290) rewrites all text content
  in-place. Preserves nav-open state, dropdown states, active indicators. Updates 30+ text slots
  across nav structure.
- **Bilingual mirror in ticker/footer**: `updateTickerLang()` (ticker.js:279-309) and footer
  re-render labels without re-injecting markup.
- **Shell controller**: `mountSharedShell().updateLang(nextLang)` calls all three in sequence,
  ensures consistency.
- **Page lang attrs**: Caller also sets `document.documentElement.lang` and `dir` (rtl for Arabic).

**Shell Guard Validation (CI)**

- **File**: `scripts/node/check-shell-guard.js` (lines 1-92). Runs in build pipeline.
- **Checks**: (1) No public HTML pages contain static `.site-nav` or `.site-footer-global` markup
  (shell must be injected). (2) `styles/partials/shell.css` uses only CSS design tokens, no raw
  color literals. (3) `NAV_DATA.en` and `NAV_DATA.ar` define exactly 7 `canonicalSurfaces`
  (homepage, tracker, calculator, countries, shops, learn, insights).
- **Fails the build** if any check fails. Prevents regressions where markup gets hardcoded or shell
  IA drifts.

**Entry Points (By Page Type)**

- **Static content pages** (`/content/*`, `/countries/*`, etc.): `bootContentPage(options)` in
  `src/lib/content-page-boot.js:23-60`. Injects nav → breadcrumbs → footer → ticker (optional) →
  related guides. Infers depth from `location.pathname` segment count.
- **Tracker/calculator** (`/tracker.html`, `/calculator.html`): `mountShell()` in
  `src/tracker/ui-shell.js:34-284`. Calls `mountSharedShell({lang, depth: 0, withSpotBar: true})`,
  wires mode tabs, overlays (alerts/planner), language toggle buttons to state mgmt.
- **Home page** (`/index.html`): `src/pages/home.js:30-40` calls `mountSharedShell()` directly, then
  mounts hero, price grid, related content.

**Shared Concerns**

- **Depth parameter**: Passed to all injectors. Used by `resolveHref()` to strip `../` from nav-data
  hrefs, then re-prepend correct number of `../` for current page depth (0 = root, 1 = /countries/,
  2 = /content/guides/, etc.).
- **No static markup**: All 390 pages have empty or minimal `<body>`. Shell is purely injected at
  runtime. Enables single-update strategy: change `nav.js`, rebuild, all pages update.
- **No shell duplication**: Guard checks + re-inject prevention means shell never appears twice even
  if script runs multiple times.

## 3. Price + FX fetch & freshness logic

**Data sources & computation:**

- Canonical gold-price payload (schema v1) → `data/gold_price.json` — written by
  `scripts/python/fetch_gold_price.py` (provider-adapter chain). Contains `xau_usd_per_oz`,
  `aed_per_gram_24k`, `timestamp_utc`, `is_fresh`, `is_fallback`, `freshness_seconds` (line 1–43).
- Last-price cache → `data/last_gold_price.json` — lightweight snapshot (only `price`,
  `posted_at_utc`, `content_hash`) written only when a fresh quote is found; used for deduplication
  and state bookkeeping.
- Provider state tracking → `data/provider_state.json` — circuit-breaker metadata per provider
  (success/failure counts, open-until timestamp, error category).

**Price formula (XAU/USD → local per gram):**

- Core math lives in `server/services/pricingEngine.js` (lines 4–87):
  `usdPerGram = (spotUsdPerOz / 31.1035) * purity` → `localPerGram = usdPerGram * fxRate`. Karat
  purity fractions: 24k=1.0, 22k≈0.917, 21k=0.875, 18k=0.75. Tola unit also supported (11.6638
  grams).
- AED peg **always** 3.6725 — hardcoded in `src/config/constants.js:9` (`CONSTANTS.AED_PEG`). Never
  fetched from API; enforced in `fxService.js:68` (line 68: `delete rates.AED` then `AED: AED_PEG`).
  Also in `normalize.py:22` (Python provider-adapter layer).

**FX rates & freshness (client-side):**

- Client fetches from `open.er-api.com/v6/latest/USD` (free tier, ~24h update cadence). Cached
  locally for 24 hours (`FX_TTL_MS = 86400000` ms in `fxService.js:15`).
- FX freshness enum: `'live'` (< 26h old), `'cached'` (24–26h), `'stale'` (> 26h). Defined in
  `src/lib/live-status.js:245–267` (`getFXFreshness`).
- Server-side FX service `server/services/fxService.js` (lines 52–88) implements same logic: tries
  live fetch with 8s timeout + 2 retries, falls back to cache, then stale cache.

**Freshness state resolution (six-state enum):**

- Single source of truth: `src/lib/live-status.js` — exported function `getLiveFreshness()` (lines
  162–232). Anti-mislabel precedence:
  1. `'unavailable'` — missing or unparseable `updatedAt`.
  2. `'fallback'` — upstream set `isFallback: true` (provider chain detected live failure; hard
     override).
  3. `'stale'` — `isFresh === false` from upstream OR age > `staleAfterMs` (75 min default, line
     27).
  4. `'cached'` — `hasLiveFailure: true` (local fetch failed) OR age is within window but
     `isFresh: true` tells us it's from cache.
  5. `'delayed'` — age > `delayedAfterMs` (30 min, line 26) but ≤ stale threshold; still trustworthy
     but UI reflects delay.
  6. `'live'` — age ≤ `delayedAfterMs`, `isFresh: true`, no local failure. **Only state** where
     "Live" label is shown.

**Timestamp & threshold logic:**

- Gold market open/close times: Sun 22:00 UTC to Fri 21:00 UTC (`GOLD_MARKET` constants in
  `live-status.js:23–28`). Market hours define whether "delayed" can be shown vs. falling back to
  "closed" badge.
- Workflow cron cadence: `gold-price-fetch.yml` runs every hour at minute :02 (lines 16–18:
  `'2 21-23 * * 0'` Sunday, `'2 * * * 1-4'` weekdays, `'2 0-20 * * 5'` Friday) **while market is
  open**. The 75-minute `STALE_AFTER_MS` tolerance (line 27 of `live-status.js`) is tightly coupled:
  one missed cron tick = still "live", two = "stale".
- Upstream flags `isFresh` and `isFallback` in `data/gold_price.json` (lines 17–18): come from
  provider-adapter (`scripts/python/fetch_gold_price.py` line 141:
  `"is_fresh": freshness_seconds <= max_fresh`).

**Freshness audit & validation:**

- Static analysis script `scripts/node/audit-freshness-coverage.js` (lines 1–437) scans all `src/`
  files to verify price-bearing surfaces use `live-status` module. Checks for literal string matches
  of the six freshness keys. Maintains exemption list (lines 96–130: e.g., "nav-data.js — category
  labels, not prices") and known-gap list (lines 136–145) for real gaps tracked separately.
- Metadata audit `scripts/node/check-freshness-metadata.js` (lines 1–97) walks the committed HTML
  pages and ensures every price-bearing element (detected by regex markers like `gold-price`,
  `karat-strip`, `hlc-price`) either has a `data-freshness-state` attribute or mounts a known
  freshness-control component (e.g., `country-status-list`).

**GitHub Actions fetch pipeline:**

- **Fetch workflow** → `.github/workflows/gold-price-fetch.yml` (lines 1–124): Orchestrates
  `scripts/python/fetch_gold_price.py` (lines 48–59). Accepts multiple provider APIs via env vars
  (`GOLD_PROVIDER_ORDER`, default: `gold_api_com,twelvedata_xauusd,fmp_gcusd`). Outputs: `fresh`
  (bool), `provider` (selected), `reason` (decision chain), `price`. On success, commits
  `data/gold_price.json`. Feeds results to `sync-price-snapshot.js` for Supabase sync (line 69).
- **Snapshot sync** → `scripts/node/sync-price-snapshot.js` (lines 1–130): Reads committed
  `data/gold_price.json`, validates schema (lines 58–77), inserts into Supabase `price_snapshots`
  table if new (lines 88–98). Duplicates detected via `raw_payload_hash` (SHA-256 of canonical JSON,
  `price-snapshots.js:54–56`). Also records `provider_runs` telemetry and updates `provider_health`
  aggregates.
- **Post to X** → `.github/workflows/post_gold.yml` (lines 1–80+): Reads cached

## 5. SEO Emission (Meta / Canonical / Hreflang / JSON-LD)

**Core Entry Point:** SEO metadata injection, validation, and sitemap generation are governed by
scripts in `scripts/node/` and `build/`, with schema templates in `src/seo/` and comprehensive
validation in `tests/`.

#### Title & Meta Description Emission

- **Static pages (`index.html`, `tracker.html`, `calculator.html`, `shops.html`, etc.)**:
  Hand-authored `<title>` and `<meta name="description">` in HTML head (e.g., `index.html:82-83`) →
  Page-specific, not generated.
- **Country/city dynamic pages** (`countries/{slug}/`, `{country}/{city}/gold-rate/`): Generated by
  `build/generatePages.js:57-120` → Title and description templated with country/city names and
  currency code; JSON-LD Dataset schema injected at build time (lines 88-90).
- **Content hub pages** (`content/guides/`, `content/faq/`): Hand-authored in HTML with `<title>`
  and meta description; schema injected post-build by `inject-schema.js` → Preserves authorial
  intent while overlaying structured data.

#### Canonical URL Management

- **Canonical builder:** `src/seo/canonical.js:12-14` → `buildCanonicalUrl()` normalizes pathnames
  and enforces HTTPS + `goldtickerlive.com` origin (no `www.`).
- **Emission:** All pages emit exactly one `<link rel="canonical" href="...">` at generation or
  injection time; validation enforces this in `scripts/node/check-seo-meta.js:89-101` (regex checks
  for count, HTTPS, and production host).
- **Redirect stubs:** Meta-refresh redirect pages (e.g., old URLs) only require canonical; hreflang
  is optional (flagged in `check-seo-meta.js:126`).
- **Canonical vs. og:url parity:** Check enforces that `og:url` meta tag matches canonical URL to
  prevent split personality in social shares and indexing (`check-seo-meta.js:119-122`).

#### Hreflang Alternates (EN/AR Pairing)

- **Builder:** `src/seo/hreflang.js:3-11` → `buildHreflangAlternates()` emits three links per page:
  - `hreflang="x-default"` → canonical EN URL
  - `hreflang="en"` → canonical EN URL
  - `hreflang="ar"` → same URL + `?lang=ar` query param
- **Emission:** All public indexable pages emit all three tags in `<head>` via
  `enforceHreflangAlternates()` (line 13-24).
- **Sitemap hreflang:** Sitemap XML also declares alternates per URL in
  `build/generateSitemap.js:52-63` (xhtml:link with rel="alternate" for x-default, en, ar).
- **Validation:** `scripts/node/check-seo-meta.js:128-136` checks all three hreflang values present
  and non-empty for every public indexable page.

#### JSON-LD Schema Injection (`inject-schema.js`)

- **Entry point:** `scripts/node/inject-schema.js` injects structured data into HTML `<head>` before
  `</head>` tag (lines 655-675).
- **Page type detection:** `detectPageType()` (lines 377-393) classifies pages (homepage, country,
  city, price, article, tool, generic) by file path patterns.
- **Schema templates by page type:**
  - **Homepage**: Organization + WebSite schemas (lines 63-100) with SearchAction target
    (`/content/search/?q={search_term_string}`).
  - **All pages except homepage**: BreadcrumbList schema (lines 106-117) built from URL path (lines
    321-367), with breadcrumb items humanized and last item URL preferring canonical (line 361).
  - **Price pages** (gold-price, gold-rate): Dataset schema (lines 196-214) describing reference
    (non-commercial) gold price data with variableMeasured labeled per karat + currency (line 486);
    also extracts FAQPage from embedded `country-page-data` JSON (lines 462-476).
  - **Content/guide pages**: Article schema (lines 123-148) with datePublished/dateModified
    preserved from existing markup (lines 498-502); FAQ articles extract FAQ panels and emit FAQPage
    schema (lines 515-520).
  - **Shops directory**: ItemList schema of JewelryStore listings (lines 230-262) extracted from
    `data/shops.js`; deliberately omits phone/geo/ratings/reviews/Offer markup (honesty contract per
    AGENTS.md) to avoid false-positive structured-data violations (lines 220-227).
  - **Calculator & Tracker tools**: WebApplication schema (lines 274-304, 541-573) describing
    feature list, free-tier Offer, and browser requirements; featureList mirrors actual UI (line
    290-296).
  - **Compare tool**: WebApplication schema (lines 577-607) for multi-country gold price comparison.
  - **Country hub** (`countries/{slug}/index.html`): Dataset schema (lines 614-625) for reference
    gold price data per currency (FAQPage added at runtime by country-page.js, not duplicated here).
- **Idempotency:** Existing JSON-LD blocks removed before re-injection (lines 643-652) so re-runs
  produce identical output; trailing whitespace collapsed (line 651).
- **Noindex exemption:** Pages with `<meta name="robots" content="...noindex...">` are skipped
  entirely (line 687).
- **Execution:** Run `node scripts/node/inject-schema.js` to inject into all pages or `--check` to
  validate existing schema (lines 778-800).

#### Sitemap Generation (`build/generateSitemap.js`)

- **Purpose:** Emits `sitemap.xml` with all indexable URLs and hreflang alternates; run
  `node build/generateSitemap.js` post-build.
- **URL discovery:** Static pages hardcoded list (lines 68-131); country/city pages iterated from
  COUNTRIES config (lines 146-178).
- **Noindex filtering:** Before emitting, checks if underlying file has noindex meta tag via
  `isNoindexFile()` (lines 16-23); if so, excludes from sitemap (lines 134-136, 149, 162, 174).
- **Hreflang in sitemap:** Each URL entry includes `xhtml:link rel="alternate"` for x-default, en
  (same URL), and ar (URL + `?lang=ar`) (lines 57-59).
- **Canonical enforcement:** Sitemap emits ONLY the canonical hub URL for each country (e.g.,
  `/countries/{slug}/` NOT `/{slug}/gold-price/`), preventing redirect-form URLs in search index
  (lines 140-145).

#### SEO Meta Regression Guard (`scripts/node/check-seo-meta.js`)

- **Purpose:** Validates canonical, hreflang, title, description, og:url, and noindex correctness on
  every HTML page; blocks build if violations found (lines 69-150).
- **Internal vs. public:** Pages in prefixes like `admin/`, `server/`, `src/`, etc. (lines 17-34)
  must be noindex; public pages must have canonical + hreflang + meta description.
- **Canonical checks:**
  - Exactly one canonical link per page (lines 90-95).
  - Must use production host `https://goldtickerlive.com/` (line 99).
  - No query string, fragment, or trailing query (lines 76-83 in tests/seo-metadata.test.js).
- **Title & description:** Public pages must have non-empty title (≥3 chars) and meta description
  (≥10 chars) (lines 104-110).
- **og:url parity:** Must exist, use production host, and match canonical URL (lines 114-123) to
  prevent split-brain metadata in social shares.
- **Hreflang:** Public pages must expose x-default, en, ar (lines 128-136); redirect stubs exempt
  (line 126).
- **Redirect stubs:** Identified by `http-equiv="refresh"` and small size (<4000 bytes) (line 41);
  only require canonical, skip hreflang (lines 126).
- **Exit code:** Process exits 1 on failure so CI catches regressions; up to 50 issues listed in
