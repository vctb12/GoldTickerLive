## Runtime console baseline (EN + AR)

Captured with `scripts/qa/capture-console-baseline.mjs` against the **production build served like
`deploy.yml`** (dist + the workflow's post-build copies of `assets/`, `data/`, `sw.js`, `styles/`,
`src/`, `favicon.svg`, `manifest.json`, `robots.txt`). 16 top-level pages × {EN default, AR
`?lang=ar`} = 32 loads. Full data: `reports/qa/console-baseline-2026-07-07.json` / `.md`.

**Headline finding: 0 uncaught page errors (`pageerror`) across all 32 EN+AR loads, and only 4
console errors total** — all 4 are `offline.html` (EN+AR) failing to load Google Tag Manager +
Microsoft Clarity, i.e. blocked external analytics in this no-egress sandbox, not a product defect.
Every other page is 0/0/0. The app throws no unhandled exceptions in either locale.

(A first capture against an **unstaged** `dist/` reported 180 console errors; every one was a
blocked external host or a missing deploy-time file — see caveats below. The staged run above is the
authoritative baseline.)

### Honest environment caveats (read before treating any console line as a defect)

1. **No outbound egress in this sandbox.** Requests to the live gold API, the FX endpoint
   (`open.er-api.com`), and Supabase (`…supabase.co/rest/v1/site_settings`) are blocked
   (`ERR_CONNECTION_RESET` / `ERR_ABORTED`). Any console error / failed-request that names an
   external host is an environment artifact, **not** a product defect — and the fact that the pages
   still render with `—` placeholders and no uncaught error is exactly the freshness-contract
   fallback behaviour working.
2. **First (discarded) capture used an unstaged `npm run build`** whose `dist/` lacks the
   deploy-time copies, so `/assets/analytics.js` and `/data/gold_price.json` 404'd. Those are
   **harness artifacts** — production copies both in (`deploy.yml`). The staged capture above
   resolves them.
3. Because egress is blocked, this run **cannot** measure the "9-min-old price still labelled Live"
   or "two different homepage prices" trust issues from `docs/revamp/00-AUDIT.md` — those need a
   live feed and are handled in Track B by code inspection + unit tests, not this capture.

### Repeatable command

```bash
npm run build
cp -r assets/. dist/assets/ && cp -r data/. dist/data/ && cp -f sw.js favicon.svg manifest.json robots.txt dist/
node scripts/qa/capture-console-baseline.mjs --serve dist --out reports/qa --stamp <label>
# or against the live site (real egress): node scripts/qa/capture-console-baseline.mjs --base https://goldtickerlive.com --pages index.html,tracker.html,...
```

## Lighthouse / pa11y note

`lighthouserc.json` (Lighthouse CI, 4 URLs) and `.pa11yci.js` (pa11y-ci, 3 URLs) are the repo's
perf/a11y gates but require network access to `temporary-public-storage` (LHCI upload) and a running
server; `pa11y-ci` is not currently installed in `node_modules`. Perf is measured in Phase 14 (with
a new offline-capable `scripts/perf/` runner) and a11y via axe-core Playwright in Phase 19 — both
locally, no external upload. This baseline records the suite + console state; Phases 14/19 add the
perf/a11y numbers.
